# Безопасность

## Почему требуется `--privileged`

SIP-exporter использует **eBPF** (extended Berkeley Packet Filter), подключённый к сокетам `AF_PACKET` для захвата SIP-трафика напрямую в ядре Linux. Для этого требуются три конкретные capabilities:

| Capability | Зачем нужна | Место в коде |
|---|---|---|
| `CAP_BPF` (или `CAP_SYS_ADMIN`) | Загрузка eBPF-программы в ядро через syscall `bpf()` | `exporter.go:120` — `ebpf.LoadCollection()` |
| `CAP_NET_RAW` | Создание raw-сокета `AF_PACKET` (`SOCK_RAW`) | `exporter.go:148` — `unix.Socket(AF_PACKET, SOCK_RAW, ...)` |
| `CAP_NET_ADMIN` | Привязка eBPF-фильтра к сокету, настройка буфера сокета | `exporter.go:186` — `SO_ATTACH_BPF`, `exporter.go:155` — `SO_RCVBUF` |

Эти capabilities доступны только root (`UID 0`), поэтому требуется `--privileged`.

### Почему `network_mode: host`

Захват пакетов через `AF_PACKET` требует прямого доступа к сетевому интерфейсу хоста. Bridge-сеть видит только виртуальный интерфейс контейнера, а не реальный SIP-трафик на физическом NIC. Обойти это нельзя — это фундаментальное требование пассивного сетевого мониторинга.

## Что контейнер делает с привилегиями

Контейнер выполняет **только чтение пакетов**:

1. **Загружает** eBPF socket filter в ядро (один раз, при запуске)
2. **Создаёт** `AF_PACKET` raw-сокет, привязанный к указанному сетевому интерфейсу
3. **Читает** пакеты из сокета в Go-канал (буфер 10 000 пакетов)
4. **Парсит** SIP-заголовки (метод, статус, Call-ID, From/To tags, CSeq, Session-Expires)
5. **Экспортирует** метрики в Prometheus через HTTP-эндпоинт `/metrics`

Всё. Никакой модификации пакетов, инъекции трафика, перенаправления сети, правил iptables/nftables или записи в файловую систему (кроме stdout/stderr для логов).

## Что контейнер НЕ делает

- **Не** модифицирует и не отбрасывает пакеты — eBPF-фильтр *пассивный* (только чтение)
- **Не** отправляет SIP-трафик — исключительно пассивный слушатель
- **Не** пишет в файловую систему хоста — все volumes подключены как `:ro` (только чтение)
- **Не** обращается к другим контейнерам, процессам или системным ресурсам
- **Не** открывает входящие порты, кроме `/metrics` на настроенном HTTP-порту (по умолчанию 2112)
- **Не** устанавливает исходящие сетевые соединения

## Минимальная атаковая поверхность

| Слой | Детали |
|---|---|
| Базовый образ | `alpine:3.22` — минимальный (~5 МБ) |
| Runtime-зависимости | `libelf` (для eBPF), `bash` (для healthcheck) |
| Приложение | Один бинарник на Go |
| Volumes | `/etc/localtime:ro`, `/etc/timezone:ro` — только чтение часового пояса |
| Сеть | Только HTTP-эндпоинт `/metrics` (порт 2112 по умолчанию) |
| Процессы | Один процесс, нет shell, нет демонов |

## Аудит eBPF-кода

Весь eBPF-фильтр — это [100 строк на C](../internal/bpf/sip.c). Он делает одну вещь: фильтрует пакеты, пропуская только UDP-трафик на настроенных SIP-портах (по умолчанию 5060/5061).

**Что делает eBPF-программа:**
1. Проверяет Ethernet-заголовок — пропускает только Ethernet-фреймы
2. Обрабатывает VLAN-теги 802.1Q — корректирует смещение при наличии
3. Фильтрует только IPv4 (`ethertype 0x0800`)
4. Валидирует длину IP-заголовка (IHL)
5. Фильтрует только UDP (`protocol 17`)
6. Читает исходный и целевой порты
7. Пропускает **только** пакеты, у которых src или dst порт совпадает с SIP/SIPS портом
8. Возвращает `skb->len` (пропустить) или `0` (не копировать в userspace — пакет всё равно доходит до адресата, он просто не копируется в пространство приложения)

**Ключевой момент:** eBPF-фильтр — это *socket filter*, а не *tc/XDP filter*. Он только управляет тем, какие пакеты копируются в буфер приложения. Отброшенные пакеты **не** теряются — они продолжают путь по сетевому стеку к адресату. Фильтр не может модифицировать или блокировать трафик.

## Отраслевой стандарт

Запуск с привилегиями для eBPF-мониторинга — стандартная практика:

| Проект | Назначение | Привилегированный? |
|---|---|---|
| [Cilium](https://github.com/cilium/cilium) | eBPF-сеть и безопасность | Да |
| [Falco](https://github.com/falcosecurity/falco) | Мониторинг системных вызовов через eBPF | Да |
| [Pixie](https://github.com/pixie-io/pixie) | eBPF-наблюдаемость Kubernetes | Да |
| [kubectl-trace](https://github.com/iovisor/kubectl-trace) | eBPF-трейсинг | Да |
| [Parca](https://github.com/parca-dev/parca) | eBPF-профилирование | Да |
| **SIP-exporter** | Мониторинг SIP-трафика через eBPF | Да |

Все eBPF-инструменты требуют `CAP_BPF` / `CAP_SYS_ADMIN` для загрузки программ в ядро. Это ограничение на уровне ядра, а не контейнера.

## Автоматическое сканирование уязвимостей

Весь код и образы контейнеров автоматически проверяются на известные уязвимости:

| Сканер | Что проверяет | Частота |
|---|---|---|
| [Go Vulncheck](https://pkg.go.dev/golang.org/x/vuln/cmd/govulncheck) | Go-зависимости по Go Vulnerability Database | Каждый push + ежедневно |
| [Trivy](https://trivy.dev) | Образ контейнера (пакеты ОС + Go-бинарники) по базам CVE | Каждый push + ежедневно |

Результаты загружаются в [вкладку Security на GitHub](https://github.com/aibudaevv/sip-exporter/security).

### Локальное сканирование

Запускайте проверки безопасности перед пушем:

```bash
make security      # govulncheck + trivy filesystem scan (быстро, без Docker build)
make trivy-image   # полная проверка образа (требует Docker build)
```

**Требования:** `govulncheck` (`go install golang.org/x/vuln/cmd/govulncheck@latest`) и `trivy` ([инструкция по установке](https://trivy.dev/latest/getting-started/installation/)).
