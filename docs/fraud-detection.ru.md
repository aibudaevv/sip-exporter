# Детекция фрода — Как это работает

> **Версия:** sip-exporter v1.3.3+
>
> sip-exporter предоставляет **сигнальную** (signal-only) детекцию фрода.
> Экспортёр не блокирует и не перехватывает трафик. Вместо этого он экспортирует
> Prometheus counter-метрики, которые инкрементируются при обнаружении
> подозрительных паттернов. Алерты настраиваются в AlertManager, а блокировка
> выполняется внешними средствами (fail2ban, правила SBC, firewall).

---

## 1. Обзор

VoIP-фрод ежегодно приносит телеком-отрасли убытки более $15 млрд (CFCA). Две
крупнейшие категории:

| Категория | Годовые убытки | Сигнал детекции |
|-----------|---------------|-----------------|
| Компрометация PBX / голосовой почты | $4.96 млрд | Сканирование регистраций + всплеск INVITE |
| Подписка / кража личности | $4.32 млрд | Смена страны регистрации |

sip-exporter покрывает обе категории четырьмя сигналами детекции:

| Сигнал | Метрика | Что детектирует |
|--------|---------|-----------------|
| Сканирование регистраций | `sip_exporter_register_scan_total` | Перечисление аккаунтов / компрометация PBX |
| Смена страны регистрации | `sip_exporter_register_country_change_total` | Перехват аккаунта из новой географии |
| Всплеск INVITE | `sip_exporter_invite_burst_total` | Начало toll-fraud / SIP-флуд DDoS |
| Утилизация сессий | `sip_exporter_sessions_utilization` | Исчерпание ёмкости / нарушение контракта |

**Важно:** Все сигналы используют лейблы `{carrier,source_country}`. IP-адрес
источника используется внутри для threshold-детектинга, но **никогда не
экспонируется** как Prometheus-лейбл (защита от взрыва кардинальности).

## 2. Сигналы детекции

### 2.1 Сканирование регистраций (Перечисление аккаунтов)

**Метрика:** `sip_exporter_register_scan_total{carrier,source_country}` (счётчик)

**Что детектирует:** Один IP-адрес успешно регистрирует множество различных SIP-аккаунтов (AOR) за короткое время — сильный индикатор компрометации PBX (массовая регистрация экстеншенов), фермы аккаунтов или атаки с украденными учётными данными для массового перехвата.

**Как это работает:**

1. При каждом успешном REGISTER (200 OK) выполняется lookup исходного IP-адреса
   запроса (IP сохраняется из REGISTER-запроса, а не из ответа — ответ приходит
   от сервера, а не от клиента).
2. AOR (`user@host`) записывается для этого IP-адреса в скользящем окне.
   Повторная регистрация того же AOR обновляет timestamp, но не увеличивает счётчик.
3. Когда количество **уникальных AOR** с одного IP достигает порога, счётчик
   инкрементируется для этого AOR и для каждого последующего нового AOR.
4. Счётчик растёт непрерывно во время сканирования, что делает `rate()`
   эффективным для алертинга.
5. Когда все AOR устаревают (выходят за пределы окна), счётчик перестаёт
   расти.

**Конфигурация:**

| Переменная окружения | По умолчанию | Описание |
|----------------------|-------------|----------|
| `SIP_EXPORTER_FRAUD_REGISTER_SCAN_THRESHOLD` | `10` | Уникальных AOR с одного IP для срабатывания |
| `SIP_EXPORTER_FRAUD_REGISTER_SCAN_WINDOW` | `60s` | Длительность скользящего окна |

**Пример:** Скомпрометированный PBX на 203.0.113.5 регистрирует 15 различных
аккаунтов за 30 секунд. При пороге=10, окне=60с:
- Регистрации 1–9 → сигнала нет (ниже порога)
- На 10-м уникальном AOR → счётчик инкрементируется (+1)
- Регистрации 11–15 → счётчик инкрементируется для каждого нового AOR (+5)
- Через 60 секунд без новых регистраций → записи устаревают, счётчик
  останавливается
- Если новый всплеск из 10+ уникальных AOR → счётчик инкрементируется снова

**Ограничения:**
- **Round-robin SBC:** SBC или регистрар, распределяющий регистрации по множеству
  различных экстеншенов от лица здоровых телефонов, может вызывать ложные
  срабатывания. Повысьте порог для легитимных регистраторов с высоким трафиком.
- **Brute-force учётных данных (401/403) этой метрикой НЕ детектируется.** Сигнал
  отслеживает только *успешные* регистрации. Для детекции brute-force используйте
  `register_failure_total{code="401"}` с преднастроенным алертом
  `SIPRegistrationBruteForce` (см. `ALERTING.md`).
- SIP-сканинг с ротацией IP-адресов (ботнет) может не достичь порога на один IP.
  Для сетевой детекции агрегируйте `register_scan_total` по всем IP в PromQL.
- **`IGNORE_OUTGOING=true` на хосте SIP-сервера:** Детекция сканирования регистраций
  требует 200 OK response для сопоставления с исходным REGISTER-запросом. При
  `SIP_EXPORTER_IGNORE_OUTGOING=true` на хосте SIP-сервера (реальный NIC, не
  SPAN/mirror) ядро фильтрует исходящие 200 OK, молча отключая этот сигнал.
  Детекция INVITE-burst не затронута (она обрабатывает только запросы).
  Используйте SPAN/mirror-порт или `IGNORE_OUTGOING=false` для включения детекции.

### 2.2 Смена страны регистрации (Перехват аккаунта)

**Метрика:** `sip_exporter_register_country_change_total{carrier,source_country}`
(счётчик)

**Что детектирует:** Один и тот же SIP-пользователь (AOR = `user@host`) успешно
перерегистрируется из другой страны — сильный индикатор перехвата аккаунта или
кражи учётных данных.

**Как это работает:**

1. При каждом успешном REGISTER (200 OK) экспортёр сохраняет регистрацию с её
   `source_country` (определяется через GeoIP или конфигурацию оператора связи).
2. Перед перезаписью предыдущей записи экспортёр сравнивает новый
   `source_country` с сохранённым.
3. Если они различаются (и предыдущий был непустым) → счётчик инкрементируется.
4. Новая страна становится базовой для будущих сравнений.

**Конфигурация не требуется** — используется существующая настройка
GeoIP/оператора связи из S003.

**Пример:** Пользователь `alice@example.com` регистрируется из России (RU).
Позже тот же AOR регистрируется из Грузии (GE):
- `source_country` меняется с `RU` на `GE` → счётчик инкрементируется
- Если `alice` снова регистрируется из `GE` → сигнала нет (та же страна)

**Ограничения:**
- **Легитимный роуминг** вызывает ложные срабатывания. Пользователь, который
  выехал за границу, сгенерирует сигнал. Это намеренно — оператор разбирается и
  закрывает, если легитимно.
- **GeoIP отключён:** Если база GeoIP не настроена и страна оператора не задана,
  `source_country` = `"unknown"` для всех → детекция не срабатывает (no-op).
  Это корректное поведение — без географических данных смену страны обнаружить
  нельзя.
- **Регистрация истекла:** Если предыдущая регистрация истекла по TTL (очистка
  удалила запись), новая регистрация из другой страны не вызовет сигнала (нет
  базовой страны для сравнения). Перехват в активной сессии (refresh из новой
  страны при живой старой регистрации) покрывается.

### 2.3 Всплеск INVITE (Toll-fraud / DDoS)

**Метрика:** `sip_exporter_invite_burst_total{carrier,source_country}` (счётчик)

**Что детектирует:** Один IP-адрес отправляет INVITE-запросы с аномально высокой
скоростью — начало toll-fraud (скомпрометированный PBX делает дорогие звонки)
 или SIP-флуд DDoS.

**Как это работает:**

1. Каждый первоначальный INVITE-запрос (не re-INVITE в существующем диалоге) с
   IP-адреса отслеживается в скользящем окне (тот же алгоритм, что для
   сканирования регистраций).
2. При достижении порога → счётчик инкрементируется для этого INVITE и для
   каждого последующего INVITE.
3. Счётчик растёт непрерывно во время всплеска, что делает `rate()`
   эффективным для алертинга.
4. При очищении окна счётчик перестаёт расти.

**Конфигурация:**

| Переменная окружения | По умолчанию | Описание |
|----------------------|-------------|----------|
| `SIP_EXPORTER_FRAUD_INVITE_BURST_THRESHOLD` | `100` | Первоначальных INVITE с одного IP для срабатывания |
| `SIP_EXPORTER_FRAUD_INVITE_BURST_WINDOW` | `60s` | Длительность скользящего окна |

**Исключение re-INVITE:** Re-INVITE внутри диалога (обновление сессии, смена
кодека, удержание вызова) исключаются из подсчёта всплеска. Учитываются только
первоначальные INVITE, начинающие новые диалоги. Это предотвращает ложные
срабатывания в легитимных колл-центрах, часто использующих re-INVITE.

**Пример:** Скомпрометированный PBX на 198.51.100.10 начинает совершать 150
звонков в минуту на платные международные номера. При пороге=100:
- INVITE 1–99 → сигнала нет (ниже порога)
- На 100-м INVITE → счётчик инкрементируется (+1)
- INVITE 101–150 → счётчик инкрементируется за каждый (+50)
- После очищения окна → счётчик останавливается; новый всплеск вызывает
  срабатывание снова

**SBC / высоконагруженные источники:** SBC (Session Border Controller), SIP-шлюз
или крупный колл-центр, мультиплексирующий множество абонентов через один IP,
может легитимно превышать порог по умолчанию (100 INVITE/мин). В этом случае
повысьте порог для данного источника или настройте `SIP_EXPORTER_IGNORE_OUTGOING=true`,
если SBC находится в вашей сети и важен только входящий фрод.

### 2.4 Утилизация сессий (Мониторинг ёмкости)

**Метрики:**
- `sip_exporter_sessions_limit{carrier}` (gauge)
- `sip_exporter_sessions_utilization{carrier}` (gauge)

**Что измеряет:** Насколько каждый оператор связи близок к настроенному лимиту
параллельных сессий. Не является сигналом фрода как таковым, но полезно для
планирования ёмкости и контроля контракта — внезапный всплеск утилизации может
указывать на фрод или неправильную настройку dialer'а.

**Как это работает:**

1. Вы настраиваете лимиты параллельных сессий для каждого оператора в YAML-файле.
2. Экспортёр экспортирует `sessions_limit{carrier}` как константный gauge из
   конфигурации.
3. При каждом scrape Prometheus экспортёр вычисляет
   `sessions_utilization = min(100, активные_сессии / лимит × 100)` по каждому
   оператору.
4. Операторы без настроенного лимита не экспортируют эти метрики.

**Capping:** Утилизация ограничена 100%. Если `активные_сессии = 300`, а
`лимит = 100`, метрика покажет 100% — как и при 101%. Это предотвращает
кардинальность / алерт-шторм, но скрывает степень превышения. Отслеживайте
`sip_exporter_sessions` (raw gauge) вместе с утилизацией для обнаружения
экстремальной перегрузки.

**`limit: 0`:** Обрабатывается как «без лимита» — оператор исключается из этих
метрик полностью (как будто его нет в списке). Значение 0 **не** означает
«0% / заблокировано».

**Конфигурация:**

| Переменная окружения | Описание |
|----------------------|----------|
| `SIP_EXPORTER_SESSIONS_LIMITS` | Путь к YAML-файлу с лимитами сессий |

**Формат YAML:**
```yaml
sessions_limits:
  - carrier: "beeline"
    limit: 500
  - carrier: "mts"
    limit: 200
  - carrier: "other"
    limit: 1000
```

## 3. Алгоритм скользящего окна

Оба rate-based сигнала используют отслеживание в скользящем окне, но с разными
внутренними структурами данных, оптимизированными для своих задач.

### Сканирование регистраций — карта с ключами AOR

Трекер сканирования дедуплицирует по AOR: каждый уникальный `user@host`
записывается один раз для каждого IP-адреса (перерегистрация того же AOR
обновляет timestamp без увеличения счётчика). Внутри используется карта с
ключами AOR.

```
REGISTER 200 OK от источника IP X
  │
  ▼
Удаление AOR с timestamp старше (now - window)
  │
  ▼
Уже сигнализировано для IP X? ─── Да ──→ пропуск (ограничение памяти)
  │
  Нет
  │
  ▼
Запись AOR в карту (обновляет timestamp если уже присутствует)
  │
  ▼
количество уникальных AOR >= threshold?
  │
  ├── Да → signaled=true, сигнал (счётчик++)
  └── Нет → ожидание новых событий
```

После сигнализации новые AOR не записываются (ограничение памяти = threshold
записей). Когда существующие AOR устаревают и счётчик опускается ниже порога,
сигнал сбрасывается — следующая регистрация возобновляет запись.

### Всплеск INVITE — слайс timestamp-ов

Трекер всплесков считает raw-скорость INVITE (дедупликация не нужна — каждый
INVITE является отдельным событием). Внутри используется сортированный слайс
timestamp-ов, обрезанный до последних `threshold+1` записей.

```
Первоначальный INVITE от источника IP X
  │
  ▼
Удаление timestamp-ов старше (now - window)
  │
  ▼
Добавление now в слайс
  │
  ▼
Обрезка слайса до последних threshold+1 записей (ограничение памяти)
  │
  ▼
len(слайса) >= threshold?
  │
  ├── Да → signaled=true, сигнал (счётчик++)
  └── Нет → ожидание новых событий
```

**Память:** Оба трекера ограничены O(threshold) записей на уникальный IP. При
стандартных порогах (scan=10, burst=100):
- Сканирование регистраций: ~10 записей AOR на IP → не более ~800 байт/IP.
- Всплеск INVITE: ~100 timestamp-ов на IP → ~2.4 КБ/IP.
- Даже при DDoS с 10 000 IP-адресов общее потребление памяти не превышает 25 МБ.

**Дедупликация сигнала:** Счётчик инкрементируется не более одного раза за
«эпизод всплеска». Эпизод заканчивается, когда все записи устаревают и выходят за
пределы окна (через eviction при каждом вызове и периодическую очистку
каждую 1 секунду). Это даёт чистый Prometheus-счётчик, пригодный для алертинга
`rate(metric[5m]) > 0`.

## 4. Примеры алертов

Эти правила соответствуют `docs/ALERTING.md` §Fraud Detection Alerts.
Канонический источник — ALERTING.md; при расхождениях править там.

> **Примечание об окне `rate()`:** `rate(counter[5m]) > 0` остаётся истинным
> ~5 минут после сигнала, т.к. окно rate включает приращение. Параметр `for: 1m`
> добавляет дополнительное требование постоянства в течение 1 минуты, снижая шум
> от кратковременных всплесков. `increase(counter[5m]) > 0` семантически
> эквивалентно для булевого алертинга — оба проверяют «было ли приращение счётчика
> в пределах окна».

```yaml
# Сканирование регистраций (перебор аккаунтов) → расследование credential stuffing
- alert: SIPRegistrationScan
  expr: rate(sip_exporter_register_scan_total[5m]) > 0
  for: 1m
  labels:
    severity: critical
  annotations:
    summary: "Обнаружена атака сканирования регистраций"
    description: "Один IP регистрирует множество разных аккаунтов на {{ $labels.carrier }} из {{ $labels.source_country }}. Возможный credential stuffing или перебор."

# Всплеск INVITE → расследование toll-fraud
- alert: SIPInviteBurst
  expr: rate(sip_exporter_invite_burst_total[5m]) > 0
  for: 1m
  labels:
    severity: critical
  annotations:
    summary: "Обнаружён всплеск INVITE"
    description: "Один IP отправляет аномально высокий поток INVITE на {{ $labels.carrier }} из {{ $labels.source_country }}. Возможный toll-fraud, DDoS или накрутка трафика."

# Смена страны регистрации → перехват аккаунта
- alert: SIPRegistrationCountryChange
  expr: sip_exporter_register_country_change_total > 0 unless on (carrier, source_country) (sip_exporter_register_country_change_total offset 5m > 0)
  for: 0m
  labels:
    severity: warning
  annotations:
    summary: "Обнаружена смена страны регистрации"
    description: "Пользователь перерегистрировался из другой страны на {{ $labels.carrier }}. Возможный перехват аккаунта."

# Исчерпание ёмкости сессий
- alert: SIPSessionCapacityExhaustion
  expr: sip_exporter_sessions_utilization > 90
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "Ёмкость сессий близка к исчерпанию"
    description: "Оператор {{ $labels.carrier }} на {{ $value | printf \"%.0f\" }}% от настроенного лимита сессий. Запланируйте расширение ёмкости или выясните причины всплеска трафика."
```

## 5. Приватность и кардинальность

| Вопрос | Обработка |
|--------|-----------|
| IP источника в лейблах | **Никогда.** IP используется только внутри. Лейблы: `{carrier,source_country}`. |
| SIP URI / номера телефонов | **Никогда** в метриках. AOR используется только как внутренний ключ map. |
| `ua_type` | Исключён из fraud-лейблов — атакующий варьирует User-Agent, что увеличивает кардинальность без пользы. |
| Неограниченный рост map IP-адресов | Ограничение скользящего окна + TTL-очистка (тик 1с). Простаивающие IP удаляются при опустошении деки. |
| Неограниченный рост AOR | Переиспользуется `registerExpiryTracker` с существующей TTL-очисткой (на основе Expires). |

## 6. Сравнение с VoIPMonitor

| Функция | sip-exporter | VoIPMonitor |
|---------|-------------|-------------|
| Сканирование регистраций | Sliding window на IP | Порог на IP |
| Смена страны | Per-AOR + GeoIP | GeoIP + ASN |
| Всплеск INVITE | Sliding window на IP | На IP + на направление |
| Toll-fraud (IRSF) | Не покрывается | Анализ стоимости направления |
| Bypass (SIM-боксы) | Не покрывается | Эвристики ACD/CLI |
| Блокировка | Только сигнал (Prometheus) | Опциональная блокировка |
| Потребление ресурсов | ~2.4 МБ на 10K IP | Полный захват CDR |

sip-exporter покрывает **топ-2 категории фрода** (компрометация PBX, кража
личности) с минимальными накладными расходами. IRSF и bypass-фрод требуют
